## PV26 아키텍처 요약

- trunk는 `model/net/trunk.py` 기준 Ultralytics YOLOv26 trunk/neck adapter다. 이번 run은 `backbone_variant=s`를 사용했고 pyramid channel은 `128 / 256 / 512`다.
- detection과 traffic-light attribute head는 각 pyramid scale 위에 붙는 convolution prediction head다.
- lane family는 `model/net/heads.py` 기준 별도 branch다. 3개 scale feature를 `SpatialFusionStem`으로 합친 뒤 query decoder로 lane / stop-line / crosswalk를 예측한다.
- query 수는 lane `24`, stop-line `8`, crosswalk `8`이다. lane은 row-anchor decoder, stop-line과 crosswalk는 spatial query decoder를 사용한다.
- 즉 구조적으로는 OD/TL과 lane family가 같은 trunk를 공유하지만, 후반 stage에서는 lane family head만 따로 미세조정할 수 있게 설계되어 있다.

## 왜 학습이 부진한가

### 1. phase 4는 최적화는 진행됐지만 일반화는 무너졌다

- phase 4는 trunk를 완전히 고정하고 lane-family head `9.19M` 파라미터만 lane-only 데이터로 미세조정했다.
- 이 stage에서 lane supervision 노출량은 매우 크다. 각 family source sample이 `1,376,256`까지 올라간다.
- 그런데 best epoch가 `1`이고, epoch `2`부터 `21`까지는 lane `TP = 0` 상태가 지속된다.
- 이건 "loss는 내려가지만 예측이 실제 metric threshold를 만족하는 방향으로는 안 가고 있다"는 뜻이다.

### 2. phase 3에서 lane family가 충분히 올라오기 전에 phase 4로 넘어갔다

- phase 3 best objective epoch 15에서도 lane `F1 = 0.0124`에 불과했다.
- stop-line과 crosswalk는 phase 3 전체에서 사실상 학습 신호가 없었다.
- 이런 상태에서 lane-only late finetune으로 넘어가면, 안정화가 아니라 오히려 불안정한 head를 더 밀어버릴 가능성이 크다.

### 3. stop-line / crosswalk는 전체 run에서 사실상 못 배웠다

- phase 1~4 전 구간에서 stop-line `F1`은 0, crosswalk `mean_polygon_iou`도 0에 머문다.
- exposure 자체는 적지 않다. phase 4에서는 stop-line / crosswalk source sample도 각 `1,376,256`이다.
- 따라서 원인은 단순 "학습 기회 부족"보다 다음 후보가 더 유력하다.
- loss weight 또는 target encoding이 너무 약하다.
- query decoder 출력이 metric threshold를 거의 못 넘긴다.
- lane-only late finetune이 실제로는 stop-line / crosswalk에 도움이 안 된다.

### 4. phase objective 설계가 붕괴를 충분히 벌주지 못한다

- 구현은 `tools/pv26_train/runtime.py`의 `_build_selection_metrics()`이고, metric 원시는 `model/engine/metrics.py`에서 온다.
- 문제는 "매칭이 0개일 때" 일부 geometry metric이 `0` 또는 missing으로 들어오고, objective 식에서는 그 값이 오히려 좋은 값처럼 쓰인다는 점이다.
- phase 4 마지막 epoch의 lane family score는 실제로 다음처럼 바닥값을 만든다.

```text
lane score
= 0.55 * f1
  + 0.20 * (1 - clamp(mean_point_distance / 40))
  + 0.15 * color_accuracy
  + 0.10 * type_accuracy
= 0.55 * 0
  + 0.20 * 1
  + 0.15 * 0
  + 0.10 * 0
= 0.20

stop_line score
= 0.55 * 0
  + 0.25 * 1
  + 0.20 * 1
= 0.45

crosswalk score
= 0.45 * 0
  + 0.35 * 0
  + 0.20 * 1
= 0.20

phase_4 objective
= 0.50 * 0.20
  + 0.30 * 0.45
  + 0.20 * 0.20
= 0.275
```

- 실제 CSV에서도 phase 4 epoch 2~21의 objective가 전부 `0.275`로 고정돼 있다.
- 즉 metric이 완전히 무너져도 objective가 0으로 떨어지지 않고 plateau floor를 유지한다.
- 그래서 phase 4는 epoch 1 이후 사실상 실패 상태였는데도, early stop은 "천천히 plateau"로만 인식하고 약 20 epoch를 더 소비했다.

## 개선 우선순위

### 우선순위 1. phase objective floor를 먼저 고쳐야 한다

- no-match 상태에서 `mean_point_distance`, `mean_angle_error`, `mean_vertex_distance`가 0 또는 missing이면 좋은 score로 해석되지 않게 바꿔야 한다.
- lane / stop-line / crosswalk component는 matched positive가 0이면 score를 강제로 `0`으로 두는 쪽이 안전하다.
- 최소한 phase 4 objective가 `0.275` 같은 인공 바닥값을 유지하지 않게 해야 한다.

### 우선순위 2. phase 4 진입 조건을 강화해야 한다

- phase 3에서 lane family가 안정적으로 non-zero인 상태가 아니면 phase 4로 넘어가지 않게 해야 한다.
- 지금 run 기준으로는 phase 3 best lane F1이 `0.0124`라서 phase 4 진입이 이르다.
- "최근 N epoch 동안 lane F1 non-zero 유지" 같은 guard가 필요하다.

### 우선순위 3. phase 4 자체를 더 약하게 시작해야 한다

- 현재 phase 4는 lane-only 샘플만 쓰고 trunk는 완전히 고정한다.
- 이 조합은 transferred state를 빠르게 망가뜨릴 위험이 크다.
- 다음 실험 후보:
- head LR를 더 낮추기
- phase 4 epoch 수와 patience를 줄이기
- lane-only 대신 phase 3 mixed sampler 일부를 유지하기
- trunk 상단 일부를 같이 열어두기

### 우선순위 4. stop-line / crosswalk는 별도 감사가 필요하다

- exposure 대비 metric이 완전히 0이므로 데이터 개수만으로 설명되지 않는다.
- target encoding, positive assignment, decoder output scale, threshold, loss weight를 따로 확인해야 한다.
- 특히 "학습은 되는데 metric threshold를 못 넘는지"와 "애초에 prediction이 거의 안 나오는지"를 분리해서 봐야 한다.
